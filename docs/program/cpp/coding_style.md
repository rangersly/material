## 头文件

通常每个 `.cc` 配套一个 `.h` 文件

常见例外包括单元测试和仅有一个 `main()` 函数的 `.cc` 文件

## 自给自足的头文件

> **自给自足** 即可以独立编译

若头文件声明了内联函数或者模板,而且文件的使用者需要实例化这些组件时,头文件必须直接或通过导入的文件间接提供这些组件的实现

头文件要有[头文件防护符](#define)

## define

> 所有头文件都应该用 `#define` 防护符来防止重复导入.

防护符格式 : `<项目>_<路径>_<文件名>_H_`

例如 : foo项目中的文件 `foo/src/bar/baz.h`

```cpp
#ifndef FOO_BAR_BAZ_H_
#define FOO_BAR_BAZ_H_
...
#endif
```

## 导入依赖

头文件应该直接导入,不要依赖间接导入

## 前向声明

**前向声明** 指没有对应定义的声明

```
class B;
void FuncInB();
extern int variable_in_b;
ABSL_DECLARE_FLAG(flag_in_b);
```

- 优点
  - 节约编译时间,避免不必要的重复编译
- 缺点
  - 隐藏了依赖关系
  - 让自动化工具难以发现定义该符号的模块
  - 修改库可能破坏前向声明
  - 引发析构点 **不完整类型** 错误

尽量避免使用

## 内联函数

> 只内联10行以下的小函数

- 优点
  - 内联函数体积小,令目标函数更加高效
  - 鼓励对短小且影响性能的函数进行内联
- 缺点
  - 滥用内联将拖慢程序
  - 内联大函数将增大体积,缓存利用率降低

> 实用经验 : 内联含有循环或 `switch` 语句的函数通常得不偿失

## include

> 推荐顺序 : 配套头文件,C系统库,C++标准库,其他库,本项目

头文件路径应相对与项目源码目录,不能出现UNIX目录别名`.`或`..`

在 `dir/foo.cc` 这个实现或测试 `dir2/foo.h` 内容的文件中,应按如下顺序导入头文件

1. `dir2/foo.h`
2. 空行
3. `c语言系统库`
4. 空行
5. `C++标准库`
6. 空行
7. 其他库
8. 空行
9. 本项目

这样的顺序可以确保在 `dir2/foo.h` 缺乏必要导入时,构建会失败

维护这些文件的人会首先发现,而不会牵连维护其他库的人

## 作用域

## 命名空间

> [!TIP]
> 应该尽可能在命名空间内放置代码
> 禁止使用using指令导出整个命名空间.禁止内联命名空间

命名空间可以有效防止全局作用域命名冲突

命名空间内的第一层代码不需要缩进,紧贴左边框,如下

```cpp
namespace mynamespace {

void myfuntion();
}
```

- 不要在std命名空间声明任何东西
- 不要前向声明标准库中的类
- 不要让头文件引入任何命名空间别名(`namespace baz = ::foo::bar::baz`)

- 鼓励使用单行嵌套命名空间

## 内部链接

> [!TIP]
> 将其他文件不需要的定义全部放入匿名命名空间(相当于static)

```cpp
namespace {
int add();
...
}
```

## 函数作用域

> [!TIP]
> 尽量将非成员函数放入命名空间,不要使用类进行分组
> 尽量不要使用全局函数

有明显的依赖关系时非成员函数更适合成为一个新类的成员

## 局部变量

> [!TIP]
> 尽可能缩小作用域

如果变量是类,放在循环外面声明通常更高效

## 静态和全局变量

> [!TIP]
> 禁止使用静态存储周期的对象变量,这会导致析构完全失效

全部使用基本类型,可以被 **平凡地析构** 的对象可以使用静态

不建议且通常禁止使用动态初始化全局变量,假如不受初始化顺序影响则允许
